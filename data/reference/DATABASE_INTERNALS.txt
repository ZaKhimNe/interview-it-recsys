## ACID Properties

- **Atomicity**
  - **Definition**: Ensures that a series of database operations within a transaction are completed entirely or not at all.
  - **Mechanism**: Implemented using transaction logs and rollback mechanisms. If any operation fails, the transaction log allows the system to revert the database to its previous state.
  - **Importance**: Critical for maintaining data integrity. Without atomicity, partial updates could lead to inconsistent data states.
  - **Common Confusion**: Atomicity is often confused with isolation. The key difference is that atomicity concerns the all-or-nothing execution of transactions, while isolation deals with concurrent transaction execution.

- **Consistency**
  - **Definition**: Ensures that a transaction brings the database from one valid state to another, maintaining database invariants.
  - **Mechanism**: Enforced through constraints, triggers, and rules defined within the database schema.
  - **Importance**: Prevents data corruption by ensuring that all transactions adhere to predefined rules.
  - **Common Confusion**: Consistency in ACID is not the same as the "C" in the CAP theorem. ACID consistency is about maintaining database rules, whereas CAP consistency involves distributed system state agreement.

- **Isolation**
  - **Definition**: Ensures that concurrent transactions do not interfere with each other.
  - **Mechanism**: Achieved through locking protocols and isolation levels (e.g., two-phase locking).
  - **Importance**: Prevents issues like dirty reads, non-repeatable reads, and phantom reads.
  - **Common Confusion**: Isolation is often mistaken for atomicity. Isolation is about managing concurrent transactions, while atomicity is about transaction completeness.

- **Durability**
  - **Definition**: Guarantees that once a transaction is committed, it remains so, even in the event of a system failure.
  - **Mechanism**: Implemented using write-ahead logging and checkpoints.
  - **Importance**: Ensures data persistence and reliability.
  - **Common Confusion**: Durability is sometimes confused with availability. Durability ensures data permanence post-commit, while availability is about system uptime.

## Transaction Isolation Levels

- **READ UNCOMMITTED**
  - **Definition**: Allows transactions to read uncommitted changes from other transactions.
  - **Issues**: Susceptible to dirty reads.
  - **Use Case**: Rarely used due to data integrity risks.

- **READ COMMITTED**
  - **Definition**: Ensures that a transaction can only read committed changes.
  - **Issues**: Prevents dirty reads but allows non-repeatable reads.
  - **Use Case**: Common default level balancing performance and consistency.

- **REPEATABLE READ**
  - **Definition**: Ensures that if a transaction reads a value, it will read the same value again.
  - **Issues**: Prevents dirty and non-repeatable reads but allows phantom reads.
  - **Use Case**: Suitable for applications requiring consistent reads within a transaction.

- **SERIALIZABLE**
  - **Definition**: The highest isolation level, ensuring complete isolation from other transactions.
  - **Issues**: Prevents all read anomalies but can lead to reduced concurrency.
  - **Use Case**: Used when absolute consistency is required.

## Indexing Deep Dive

- **B-tree Structure**
  - **Definition**: A balanced tree data structure that maintains sorted data and allows searches, sequential access, insertions, and deletions in logarithmic time.
  - **Mechanism**: Nodes contain keys and pointers, with leaf nodes containing actual data pointers.
  - **Importance**: Provides efficient data retrieval with O(log n) complexity.
  - **Common Confusion**: B-trees are often confused with binary trees. B-trees have multiple children per node, optimizing disk read operations.

- **Hash Index**
  - **Definition**: Uses a hash function to map keys to locations in a hash table.
  - **Mechanism**: Directly maps keys to data locations, providing O(1) average time complexity for lookups.
  - **Importance**: Ideal for equality searches but not suitable for range queries.
  - **Common Confusion**: Hash indexes are not suitable for range queries, unlike B-trees.

- **Composite Index Column Order**
  - **Definition**: An index on multiple columns, where column order affects query performance.
  - **Mechanism**: The order should match the most common query patterns.
  - **Importance**: Improves query performance by optimizing index usage.
  - **Common Confusion**: Misordering columns can lead to inefficient index usage.

- **Covering Index**
  - **Definition**: An index that contains all the columns needed to satisfy a query.
  - **Mechanism**: Reduces the need for accessing the table, as all data is in the index.
  - **Importance**: Enhances performance by minimizing I/O operations.
  - **Common Confusion**: Not all indexes are covering; they must include all queried columns.

- **Index Selectivity**
  - **Definition**: The ratio of unique indexed values to the total number of rows.
  - **Mechanism**: High selectivity indicates a more efficient index.
  - **Importance**: Affects the decision to use an index in query execution.
  - **Common Confusion**: Low selectivity indexes may not improve performance significantly.

## Query Execution Plan

- **Seq Scan vs Index Scan**
  - **Seq Scan**: Scans the entire table, used when no suitable index exists.
  - **Index Scan**: Uses an index to locate data, more efficient for selective queries.
  - **Importance**: Understanding the plan helps optimize query performance.
  - **Common Confusion**: Assuming index scans are always faster; depends on query selectivity.

- **Cost Estimation**
  - **Definition**: The database's prediction of the resources required to execute a query.
  - **Mechanism**: Based on statistics like table size and index selectivity.
  - **Importance**: Guides query optimization decisions.
  - **Common Confusion**: Cost is not always directly proportional to execution time.

## Database Scaling

- **Vertical vs Horizontal Scaling**
  - **Vertical Scaling**: Adding resources to a single node (e.g., CPU, RAM).
  - **Horizontal Scaling**: Adding more nodes to distribute load.
  - **Importance**: Determines the scalability and performance of a database system.
  - **Common Confusion**: Vertical scaling has hardware limits, while horizontal scaling requires data distribution strategies.

- **Read Replicas**
  - **Definition**: Copies of the primary database used to distribute read traffic.
  - **Mechanism**: Asynchronous replication from the primary database.
  - **Importance**: Improves read performance and availability.
  - **Common Confusion**: Read replicas do not improve write performance.

- **Sharding Strategies: Range/Hash**
  - **Range Sharding**: Divides data based on key ranges.
  - **Hash Sharding**: Distributes data using a hash function.
  - **Importance**: Affects data distribution and query performance.
  - **Common Confusion**: Range sharding can lead to uneven data distribution, while hash sharding provides more balanced distribution.

## CAP Theorem Trade-offs

- **Consistency, Availability, Partition Tolerance**
  - **Definition**: A distributed system can only guarantee two of the three properties at any time.
  - **Mechanism**: Trade-offs depend on system requirements (e.g., CP vs AP systems).
  - **Importance**: Guides design decisions in distributed databases.
  - **Common Confusion**: Misunderstanding that CAP means choosing only two properties; it's about prioritizing under failure conditions.

## Connection Pooling

- **Definition**: A technique to manage database connections efficiently by reusing existing connections.
- **Mechanism**: Maintains a pool of open connections for reuse, reducing the overhead of opening new connections.
- **Importance**: Enhances performance by minimizing connection latency and resource usage.
- **Common Confusion**: Believing connection pooling increases the number of concurrent connections; it optimizes existing connections instead.

**Interviewer Red Flags**:
- Confusing ACID properties with CAP theorem properties.
- Misunderstanding the implications of different isolation levels, particularly the differences between non-repeatable reads and phantom reads.
- Incorrectly assuming that all indexes improve query performance without considering selectivity and query patterns.
