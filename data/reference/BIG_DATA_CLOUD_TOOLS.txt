## Apache Spark

- **RDD vs DataFrame vs Dataset**
  - **RDD (Resilient Distributed Dataset)**: The fundamental data structure of Spark, RDDs are immutable distributed collections of objects. They support two types of operations: transformations (e.g., map, filter) and actions (e.g., collect, count). RDDs are type-safe and can handle any type of data.
  - **DataFrame**: A distributed collection of data organized into named columns, similar to a table in a relational database. DataFrames are optimized for performance using Spark's Catalyst optimizer and Tungsten execution engine. They are not type-safe.
  - **Dataset**: A combination of RDD and DataFrame, providing the benefits of both. Datasets are type-safe and use Catalyst for optimization. They are ideal for complex operations that require compile-time type checking.
  - **Common Confusion**: Many candidates confuse DataFrames with Datasets. The key difference is that Datasets are type-safe, whereas DataFrames are not.

- **Lazy Evaluation**
  - Spark defers execution of transformations until an action is called. This allows Spark to optimize the execution plan by combining transformations and reducing data shuffling.
  - **Why It Matters**: Lazy evaluation improves performance by minimizing data movement and optimizing resource usage.

- **Transformations vs Actions**
  - **Transformations**: Operations that create a new RDD/DataFrame/Dataset from an existing one (e.g., map, filter). They are lazy and do not trigger computation.
  - **Actions**: Operations that trigger computation and return a result to the driver program (e.g., collect, count).
  - **Red Flag**: Confusing transformations with actions. Transformations do not execute until an action is called.

- **Wide vs Narrow Transformations**
  - **Narrow Transformations**: Operations where each input partition contributes to only one output partition (e.g., map, filter). They do not require data shuffling.
  - **Wide Transformations**: Operations where input partitions contribute to multiple output partitions (e.g., groupByKey, reduceByKey). They require data shuffling.
  - **Why It Matters**: Wide transformations are more expensive due to the shuffling of data across the network.

- **Shuffle Operations**
  - Involve redistributing data across partitions, which is costly in terms of performance. Common shuffle operations include groupByKey and join.
  - **Optimization**: Minimize shuffles by using operations like reduceByKey instead of groupByKey when possible.

- **Partitioning Strategy**
  - Determines how data is distributed across partitions. Proper partitioning can significantly improve performance by reducing shuffling and ensuring balanced workloads.
  - **Red Flag**: Ignoring partitioning can lead to skewed data distribution and inefficient resource utilization.

- **Broadcast Join vs Shuffle Join**
  - **Broadcast Join**: Efficient for joining a large dataset with a small one. The smaller dataset is broadcast to all nodes, avoiding shuffling.
  - **Shuffle Join**: Used when both datasets are large. It involves shuffling data across the network, which is costly.
  - **Why It Matters**: Choosing the wrong join strategy can lead to performance bottlenecks.

- **Spark SQL**
  - Allows querying of structured data using SQL syntax. It leverages the Catalyst optimizer for query optimization and supports integration with Hive.
  - **Predicate Pushdown/Columnar Storage**: Optimizations that reduce the amount of data read from disk by applying filters early and storing data in a columnar format for efficient access.

## Apache Kafka

- **Topics**
  - A category or feed name to which records are published. Topics are partitioned for parallelism and scalability.

- **Partitions**
  - Each topic is split into partitions, which are the unit of parallelism in Kafka. Partitions allow for load balancing and fault tolerance.

- **Replication Factor**
  - The number of copies of a partition maintained across the Kafka cluster. A higher replication factor increases fault tolerance but requires more storage.

- **Consumer Groups**
  - A group of consumers that work together to consume messages from a topic. Each partition is consumed by only one consumer within a group, ensuring parallel processing.

- **Offset Management**
  - Kafka tracks the position of a consumer in a partition using offsets. Consumers can commit offsets to track their progress.

- **Exactly-Once Semantics**
  - Ensures that messages are neither lost nor duplicated, even in the presence of failures. Achieved through idempotent producers and transactional writes.

## Parquet vs Avro vs ORC

- **Columnar vs Row-Based Storage**
  - **Parquet**: A columnar storage format optimized for complex nested data structures. Supports efficient compression and encoding schemes.
  - **Avro**: A row-based storage format that is compact and fast for serialization. Ideal for write-heavy workloads.
  - **ORC**: A columnar storage format similar to Parquet, optimized for read-heavy workloads with advanced compression techniques.
  - **Common Confusion**: Confusing columnar with row-based formats. Columnar formats are better for analytical queries, while row-based formats are better for transactional workloads.

## Docker Containerization

- **Docker**: A platform for developing, shipping, and running applications in containers. Containers are lightweight, portable, and consistent across environments.
- **Why It Matters**: Docker simplifies deployment and scaling of applications, ensuring consistency across development, testing, and production environments.
- **Red Flag**: Confusing containers with virtual machines. Containers share the host OS kernel, making them more lightweight than VMs.

## S3/GCS Object Storage Patterns

- **Object Storage**: A storage architecture that manages data as objects, typically used for storing unstructured data.
- **S3 (Amazon Simple Storage Service)** and **GCS (Google Cloud Storage)**: Provide scalable, durable, and secure object storage solutions.
- **Patterns**:
  - **Data Lake**: Using S3/GCS as a central repository for raw and processed data.
  - **Versioning**: Maintaining multiple versions of objects for data recovery and auditing.
  - **Lifecycle Policies**: Automating data retention and deletion to manage storage costs.
- **Why It Matters**: Proper use of object storage patterns can optimize cost, performance, and data management in cloud environments.
