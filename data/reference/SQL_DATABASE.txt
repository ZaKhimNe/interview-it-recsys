## SQL Fundamentals

- **SQL (Structured Query Language)**: A domain-specific language used for managing and manipulating relational databases.
  - **Mechanism**: SQL allows users to query, insert, update, and delete data. It also provides commands for schema creation and modification.
  - **Importance**: Fundamental for data retrieval and manipulation in relational databases. Mastery of SQL is crucial for data analysis and engineering tasks.
  - **Common Confusions**: Confusing SQL with NoSQL. SQL is used for structured data with predefined schema, while NoSQL is used for unstructured or semi-structured data.

## JOINs

- **INNER JOIN**: Combines rows from two or more tables based on a related column, returning only matching rows.
  - **Mechanism**: Uses a comparison operator to match rows between tables.
  - **Importance**: Essential for combining data from multiple tables where relationships exist.
  - **Red Flag**: Misunderstanding that INNER JOIN returns all rows from both tables.

- **LEFT JOIN (LEFT OUTER JOIN)**: Returns all rows from the left table and matched rows from the right table. Unmatched rows from the right table return NULL.
  - **Mechanism**: Preserves all rows from the left table regardless of matches.
  - **Importance**: Useful for queries where all records from one table are needed, regardless of matches.
  - **Red Flag**: Confusing LEFT JOIN with INNER JOIN.

- **RIGHT JOIN (RIGHT OUTER JOIN)**: Returns all rows from the right table and matched rows from the left table. Unmatched rows from the left table return NULL.
  - **Mechanism**: Preserves all rows from the right table.
  - **Importance**: Less common but useful when the right table's data is prioritized.
  - **Red Flag**: Assuming RIGHT JOIN is the same as LEFT JOIN.

- **FULL JOIN (FULL OUTER JOIN)**: Returns all rows when there is a match in either left or right table records.
  - **Mechanism**: Combines results of both LEFT and RIGHT JOINs.
  - **Importance**: Useful for a complete dataset from both tables, including unmatched rows.
  - **Red Flag**: Misunderstanding that FULL JOIN only returns matched rows.

## Subqueries

- **Subquery**: A query nested inside another query.
  - **Mechanism**: Can be used in SELECT, INSERT, UPDATE, or DELETE statements.
  - **Importance**: Allows complex queries by breaking them into simpler parts.
  - **Common Confusions**: Misusing subqueries where JOINs would be more efficient.

## Window Functions

- **ROW_NUMBER()**: Assigns a unique sequential integer to rows within a partition of a result set.
  - **Mechanism**: Resets the number for each partition.
  - **Importance**: Useful for pagination and ranking.
  - **Red Flag**: Confusing ROW_NUMBER() with RANK().

- **RANK()**: Assigns a rank to each row within a partition, with gaps in ranking for ties.
  - **Mechanism**: Rows with equal values receive the same rank, and the next rank is incremented by the number of tied rows.
  - **Importance**: Useful for ranking without dense ranking.
  - **Red Flag**: Assuming RANK() behaves like ROW_NUMBER().

- **DENSE_RANK()**: Similar to RANK(), but without gaps in ranking.
  - **Mechanism**: Rows with equal values receive the same rank, and the next rank is incremented by one.
  - **Importance**: Useful when a continuous ranking is needed.
  - **Red Flag**: Confusing DENSE_RANK() with RANK().

- **LAG()**: Accesses data from a previous row in the same result set.
  - **Mechanism**: Allows backward data retrieval within a partition.
  - **Importance**: Useful for comparing current and previous row values.
  - **Red Flag**: Misunderstanding LAG() as a forward-looking function.

- **LEAD()**: Accesses data from a subsequent row in the same result set.
  - **Mechanism**: Allows forward data retrieval within a partition.
  - **Importance**: Useful for comparing current and next row values.
  - **Red Flag**: Confusing LEAD() with LAG().

- **NTILE()**: Distributes rows into a specified number of approximately equal groups.
  - **Mechanism**: Assigns a bucket number to each row.
  - **Importance**: Useful for dividing data into quantiles.
  - **Red Flag**: Misunderstanding NTILE() as a ranking function.

## CTEs (Common Table Expressions)

- **CTE**: A temporary result set that can be referenced within a SELECT, INSERT, UPDATE, or DELETE statement.
  - **Mechanism**: Defined using the WITH clause.
  - **Importance**: Improves query readability and organization.
  - **Common Confusions**: Confusing CTEs with subqueries; CTEs can be recursive and are more readable.

## Indexes

- **B-tree Index**: A balanced tree data structure that maintains sorted data and allows searches, sequential access, insertions, and deletions in logarithmic time.
  - **Mechanism**: Nodes contain keys and pointers to child nodes.
  - **Importance**: Speeds up data retrieval operations.
  - **Complexity**: O(log n) for lookup.
  - **Red Flag**: Assuming indexes always improve performance without considering overhead.

- **Composite Index**: An index on multiple columns.
  - **Mechanism**: Useful for queries filtering on multiple columns.
  - **Importance**: Improves performance of multi-column queries.
  - **Red Flag**: Misunderstanding that column order in composite indexes is irrelevant.

- **Covering Index**: An index that contains all the columns needed for a query.
  - **Mechanism**: Eliminates the need to access the table data.
  - **Importance**: Reduces I/O operations and improves query performance.
  - **Red Flag**: Assuming covering indexes are always beneficial without considering storage costs.

## Query Optimization

- **Query Optimization**: The process of modifying a query to improve performance.
  - **Mechanism**: Involves rewriting queries, using indexes, and analyzing execution plans.
  - **Importance**: Critical for efficient database performance.
  - **Common Confusions**: Assuming query optimization is only about adding indexes.

## Execution Plan

- **Execution Plan**: A detailed roadmap of how a SQL query will be executed by the database engine.
  - **Mechanism**: Shows the steps and order of operations.
  - **Importance**: Helps identify performance bottlenecks and optimize queries.
  - **Red Flag**: Ignoring execution plans when diagnosing slow queries.

## Normalization

- **1NF (First Normal Form)**: Ensures that each column contains atomic values and each record is unique.
  - **Mechanism**: Eliminates repeating groups.
  - **Importance**: Reduces redundancy and improves data integrity.
  - **Red Flag**: Confusing atomic values with non-atomic values.

- **2NF (Second Normal Form)**: Builds on 1NF by removing partial dependencies.
  - **Mechanism**: Ensures that non-key attributes are fully functionally dependent on the primary key.
  - **Importance**: Further reduces redundancy.
  - **Red Flag**: Assuming 2NF is achieved without 1NF.

- **3NF (Third Normal Form)**: Builds on 2NF by removing transitive dependencies.
  - **Mechanism**: Ensures that non-key attributes are not dependent on other non-key attributes.
  - **Importance**: Ensures data integrity and reduces redundancy.
  - **Red Flag**: Confusing transitive dependencies with partial dependencies.

## Database Design

- **Database Design**: The process of defining the structure of a database, including tables, relationships, and constraints.
  - **Mechanism**: Involves normalization, defining keys, and establishing relationships.
  - **Importance**: Ensures efficient data storage, retrieval, and integrity.
  - **Common Confusions**: Assuming database design is only about normalization; it also involves understanding business requirements and data usage patterns.
