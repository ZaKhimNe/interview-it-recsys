## Data Warehouse vs Data Lake vs Lakehouse

- **Data Warehouse**
  - **Definition**: A centralized repository designed for query and analysis, optimized for structured data.
  - **Mechanism**: Utilizes schema-on-write, meaning data is structured before storage. Typically uses OLAP (Online Analytical Processing) for fast query performance.
  - **Practical Importance**: Ideal for business intelligence and reporting where data consistency and speed are critical.
  - **Common Confusions**: Often confused with data lakes. Key difference is that data warehouses require structured data upfront, while data lakes can store raw data.
  - **Red Flags**: Misunderstanding the need for ETL (Extract, Transform, Load) processes in data warehouses.

- **Data Lake**
  - **Definition**: A storage repository that holds a vast amount of raw data in its native format until needed.
  - **Mechanism**: Uses schema-on-read, allowing for flexibility in data storage and retrieval. Supports a variety of data types including structured, semi-structured, and unstructured data.
  - **Practical Importance**: Useful for data scientists and analysts who need to perform exploratory data analysis on large datasets.
  - **Common Confusions**: Confused with data warehouses due to the term "data storage". The key difference is the flexibility and variety of data types supported.
  - **Red Flags**: Assuming data lakes automatically provide data quality and governance.

- **Lakehouse**
  - **Definition**: A hybrid architecture that combines the benefits of data lakes and data warehouses.
  - **Mechanism**: Supports ACID transactions and schema enforcement using formats like Delta Lake.
  - **Practical Importance**: Provides the flexibility of data lakes with the reliability and performance of data warehouses.
  - **Common Confusions**: Misunderstanding the role of Delta format in enabling ACID transactions.
  - **Red Flags**: Failing to recognize the importance of ACID compliance in maintaining data integrity.

## Dimensional Modeling

- **Fact Tables**
  - **Definition**: Central tables in a star schema that store quantitative data for analysis.
  - **Types of Measures**:
    - **Additive**: Measures that can be summed across all dimensions (e.g., sales revenue).
    - **Semi-additive**: Measures that can be summed across some dimensions but not all (e.g., account balance).
    - **Non-additive**: Measures that cannot be summed (e.g., ratios).
  - **Practical Importance**: Understanding the type of measure is crucial for accurate data aggregation and reporting.
  - **Red Flags**: Misclassifying measures, leading to incorrect data analysis.

- **Dimension Tables**
  - **Definition**: Tables that store attributes related to dimensions of the data (e.g., time, geography).
  - **Mechanism**: Typically contain descriptive attributes and use surrogate keys for joins.
  - **Practical Importance**: Enable slicing and dicing of data for detailed analysis.
  - **Red Flags**: Using natural keys instead of surrogate keys, which can lead to performance issues.

- **Surrogate Keys**
  - **Definition**: Artificial keys used in dimension tables to uniquely identify records.
  - **Mechanism**: Typically integer-based, providing a consistent and efficient way to join tables.
  - **Practical Importance**: Essential for maintaining data integrity and performance in joins.
  - **Red Flags**: Confusing surrogate keys with natural keys, leading to potential data integrity issues.

## Star Schema vs Snowflake Schema Trade-offs

- **Star Schema**
  - **Definition**: A simple database schema with a central fact table connected to dimension tables.
  - **Mechanism**: Denormalized structure, which simplifies queries and improves performance.
  - **Practical Importance**: Preferred for its simplicity and speed in query execution.
  - **Red Flags**: Assuming star schema is always better without considering data redundancy.

- **Snowflake Schema**
  - **Definition**: An extension of the star schema where dimension tables are normalized.
  - **Mechanism**: Normalized structure reduces data redundancy but can complicate queries.
  - **Practical Importance**: Useful for complex data models where data redundancy is a concern.
  - **Red Flags**: Over-normalizing, leading to complex and slow queries.

## SCD Type 1/2/3 Implementation

- **SCD Type 1**
  - **Definition**: Overwrites old data with new data, maintaining only current values.
  - **Practical Importance**: Simple to implement but loses historical data.
  - **Red Flags**: Using Type 1 when historical data tracking is required.

- **SCD Type 2**
  - **Definition**: Tracks historical data by creating multiple records with versioning.
  - **Practical Importance**: Essential for maintaining historical accuracy in data analysis.
  - **Red Flags**: Failing to implement versioning correctly, leading to data inaccuracies.

- **SCD Type 3**
  - **Definition**: Stores previous and current data in the same record, typically using additional columns.
  - **Practical Importance**: Useful for tracking limited historical changes.
  - **Red Flags**: Misunderstanding the limited historical tracking capability of Type 3.

## Data Vault (Hub/Link/Satellite)

- **Hub**
  - **Definition**: Central tables that store unique business keys.
  - **Mechanism**: Provides a consistent and centralized point of reference for data.
  - **Practical Importance**: Ensures data integrity and consistency across the data model.
  - **Red Flags**: Confusing hubs with fact tables, leading to incorrect data modeling.

- **Link**
  - **Definition**: Tables that capture relationships between hubs.
  - **Mechanism**: Facilitates many-to-many relationships and historical tracking.
  - **Practical Importance**: Essential for complex data relationships and historical analysis.
  - **Red Flags**: Misunderstanding the role of links in capturing relationships.

- **Satellite**
  - **Definition**: Tables that store descriptive attributes related to hubs and links.
  - **Mechanism**: Allows for flexible and scalable attribute storage.
  - **Practical Importance**: Enables detailed data analysis without impacting core data structures.
  - **Red Flags**: Overloading satellites with unnecessary attributes, leading to performance issues.

## Lambda Architecture (Batch Layer + Speed Layer)

- **Batch Layer**
  - **Definition**: Processes large volumes of data in batch mode for comprehensive analysis.
  - **Mechanism**: Utilizes distributed processing frameworks like Hadoop.
  - **Practical Importance**: Provides accurate and complete data views for historical analysis.
  - **Red Flags**: Ignoring latency issues inherent in batch processing.

- **Speed Layer**
  - **Definition**: Processes real-time data for immediate insights.
  - **Mechanism**: Utilizes stream processing frameworks like Apache Kafka.
  - **Practical Importance**: Provides low-latency access to real-time data.
  - **Red Flags**: Assuming speed layer can replace batch layer for all use cases.

## Kappa Architecture

- **Definition**: A streamlined architecture that processes data in real-time without a separate batch layer.
- **Mechanism**: Relies entirely on stream processing for both historical and real-time data.
- **Practical Importance**: Simplifies architecture and reduces latency for real-time applications.
- **Red Flags**: Misunderstanding the trade-offs in data accuracy and completeness compared to Lambda architecture.

## Data Mesh (Domain Ownership, Data as Product)

- **Domain Ownership**
  - **Definition**: Decentralized data management where domain teams own their data.
  - **Mechanism**: Empowers teams to manage and govern their data independently.
  - **Practical Importance**: Enhances agility and scalability in data management.
  - **Red Flags**: Assuming domain ownership eliminates the need for centralized governance.

- **Data as Product**
  - **Definition**: Treating data as a product with clear ownership, quality standards, and lifecycle management.
  - **Mechanism**: Involves defining clear SLAs and quality metrics for data products.
  - **Practical Importance**: Ensures data quality and reliability for end-users.
  - **Red Flags**: Neglecting the importance of data product management in ensuring data quality.

## Partitioning Strategies (By Date, By Key)

- **By Date**
  - **Definition**: Partitioning data based on time intervals (e.g., daily, monthly).
  - **Mechanism**: Facilitates efficient time-based queries and data management.
  - **Practical Importance**: Essential for time-series data and historical analysis.
  - **Red Flags**: Over-partitioning, leading to excessive small files and performance degradation.

- **By Key**
  - **Definition**: Partitioning data based on specific keys (e.g., customer ID).
  - **Mechanism**: Distributes data evenly across partitions for balanced query performance.
  - **Practical Importance**: Useful for evenly distributed workloads and parallel processing.
  - **Red Flags**: Poor key selection leading to data skew and unbalanced partitions.
