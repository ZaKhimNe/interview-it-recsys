## Microservices vs Monolithic

- **Microservices Architecture**
  - **Definition**: An architectural style that structures an application as a collection of loosely coupled services, each implementing a specific business capability.
  - **Underlying Mechanism**: Each service can be developed, deployed, and scaled independently, often using different programming languages and databases.
  - **When/Why It Matters**: Ideal for large, complex applications requiring frequent updates and scalability. Allows teams to work on different services simultaneously without affecting others.
  - **Trade-offs**: 
    - **Deployment Independence**: Facilitates continuous deployment and integration. Services can be updated without redeploying the entire application.
    - **Network Overhead**: Increased inter-service communication can lead to latency and complexity in managing network calls.
  - **Common Confusions**: Many candidates confuse microservices with SOA (Service-Oriented Architecture). The key difference is that microservices are more granular and independently deployable.

- **Monolithic Architecture**
  - **Definition**: A traditional unified model for designing software where all components are interconnected and interdependent.
  - **Underlying Mechanism**: The application is a single unit, often leading to a single codebase and deployment.
  - **When/Why It Matters**: Suitable for small to medium-sized applications with less complexity and fewer scalability requirements.
  - **Trade-offs**:
    - **Simplicity**: Easier to develop and deploy initially, with a single codebase.
    - **Scalability Limitations**: Scaling requires deploying the entire application, which can be inefficient.

## REST API Design

- **Statelessness**
  - **Definition**: Each HTTP request from a client to server must contain all the information the server needs to fulfill that request.
  - **Underlying Mechanism**: No client context is stored on the server between requests.
  - **When/Why It Matters**: Enhances scalability and reliability. Each request is independent, simplifying server design.

- **Idempotency**
  - **Definition**: An operation that can be applied multiple times without changing the result beyond the initial application.
  - **Underlying Mechanism**: Ensures that repeated requests have the same effect as a single request.
  - **When/Why It Matters**: Critical for safe retries in network communication. PUT and DELETE methods should be idempotent.

- **HTTP Methods**
  - **GET**: Retrieve data. Safe and idempotent.
  - **POST**: Create new resources. Not idempotent.
  - **PUT**: Update resources. Idempotent.
  - **DELETE**: Remove resources. Idempotent.
  - **Common Confusions**: Confusing POST with PUT. POST creates new resources, while PUT updates existing ones.

- **Status Codes**
  - **2xx**: Success (e.g., 200 OK, 201 Created)
  - **4xx**: Client errors (e.g., 400 Bad Request, 404 Not Found)
  - **5xx**: Server errors (e.g., 500 Internal Server Error)

- **Versioning**
  - **Definition**: Managing changes to an API without disrupting existing clients.
  - **Mechanism**: Use URI versioning (e.g., /v1/resource) or header versioning.
  - **When/Why It Matters**: Ensures backward compatibility and smooth transition between API versions.

## Message Queues

- **Kafka vs RabbitMQ**
  - **Kafka**
    - **Use Cases**: High-throughput, real-time data streaming, and log aggregation.
    - **Mechanism**: Distributed commit log, partitioned for scalability.
    - **Common Confusions**: Kafka is often mistaken for a traditional message queue. It is more of a distributed streaming platform.
  - **RabbitMQ**
    - **Use Cases**: Complex routing, message delivery guarantees, and lower latency.
    - **Mechanism**: Broker-based, supports multiple messaging protocols.
    - **Common Confusions**: RabbitMQ is sometimes confused with Kafka for high-throughput scenarios, but it excels in message delivery guarantees.

- **Pub/Sub vs Point-to-Point**
  - **Pub/Sub**: Messages are broadcast to all subscribers. Suitable for event-driven architectures.
  - **Point-to-Point**: Messages are sent to a single consumer. Suitable for task distribution.

## Caching Strategies

- **Cache-aside**
  - **Definition**: The application code is responsible for loading data into the cache.
  - **Mechanism**: If data is not in the cache, the application fetches it from the database and stores it in the cache.
  - **When/Why It Matters**: Provides flexibility but requires more application logic.

- **Write-through**
  - **Definition**: Data is written to the cache and the database simultaneously.
  - **Mechanism**: Ensures cache consistency with the database.
  - **When/Why It Matters**: Simplifies read operations but can introduce write latency.

- **Write-behind**
  - **Definition**: Data is written to the cache and asynchronously to the database.
  - **Mechanism**: Reduces write latency but risks data loss on failure.
  - **When/Why It Matters**: Suitable for high-write throughput scenarios.

- **TTL (Time-to-Live)**
  - **Definition**: Expiration time for cached data.
  - **Mechanism**: Automatically invalidates cache entries after a specified period.
  - **When/Why It Matters**: Prevents stale data but requires careful tuning.

- **Redis Data Structures**
  - **Strings, Lists, Sets, Hashes, Sorted Sets**: Each has specific use cases and performance characteristics.
  - **Common Confusions**: Misusing data structures can lead to inefficient memory usage and performance issues.

## Load Balancing

- **Round-robin**
  - **Definition**: Distributes requests evenly across servers.
  - **Mechanism**: Cycles through a list of servers.
  - **When/Why It Matters**: Simple and effective for evenly distributed workloads.

- **Least-connections**
  - **Definition**: Directs traffic to the server with the fewest active connections.
  - **Mechanism**: Balances load based on current server utilization.
  - **When/Why It Matters**: Useful for uneven request loads.

- **Consistent Hashing**
  - **Definition**: Distributes requests based on a hash of the request data.
  - **Mechanism**: Minimizes re-distribution of requests when servers are added or removed.
  - **When/Why It Matters**: Critical for distributed caching systems.

## High Availability Patterns

- **Failover**
  - **Definition**: Automatic switching to a standby system upon failure.
  - **Mechanism**: Uses redundant systems to ensure continuity.
  - **When/Why It Matters**: Essential for mission-critical applications.

- **Circuit Breaker**
  - **Definition**: Prevents repeated requests to a failing service.
  - **Mechanism**: Opens a circuit after failures, allowing recovery time.
  - **When/Why It Matters**: Protects systems from cascading failures.

- **Bulkhead**
  - **Definition**: Isolates components to prevent failure propagation.
  - **Mechanism**: Limits resource usage per component.
  - **When/Why It Matters**: Enhances system resilience by containing failures.

## System Design for Data-Intensive Applications

- **Scalability**
  - **Definition**: Ability to handle increased load by adding resources.
  - **Mechanism**: Horizontal scaling (adding more nodes) vs. vertical scaling (adding more power to existing nodes).
  - **When/Why It Matters**: Critical for applications with growing data and user demands.

- **Data Partitioning**
  - **Definition**: Dividing data across multiple databases or tables.
  - **Mechanism**: Sharding based on keys or ranges.
  - **When/Why It Matters**: Improves performance and manageability of large datasets.

- **Data Consistency**
  - **Definition**: Ensuring data accuracy and uniformity across systems.
  - **Mechanism**: Strong vs. eventual consistency models.
  - **When/Why It Matters**: Balancing consistency with availability and partition tolerance (CAP theorem).

- **Interviewer Red Flags**
  - Confusing microservices with monolithic architecture.
  - Misunderstanding REST API idempotency.
  - Incorrectly applying caching strategies, such as using write-behind without understanding data loss risks.
