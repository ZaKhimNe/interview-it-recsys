## ETL vs ELT Differences and When to Use Each

- **ETL (Extract, Transform, Load)**:
  - **Definition**: A data processing pipeline where data is extracted from source systems, transformed into a suitable format, and then loaded into a destination system.
  - **Underlying Mechanism**: Data is transformed before loading, often using staging areas or intermediate storage. This approach is beneficial when data needs to be cleaned or aggregated before analysis.
  - **When to Use**: Ideal for traditional data warehousing environments where data integrity and consistency are critical before loading. Suitable for complex transformations and when the destination system has limited processing capabilities.
  - **Common Confusion**: Many candidates confuse ETL with ELT. The key difference is the order of operations—ETL transforms data before loading, while ELT loads data before transforming.

- **ELT (Extract, Load, Transform)**:
  - **Definition**: A data processing pipeline where data is extracted from source systems, loaded into a destination system, and then transformed.
  - **Underlying Mechanism**: Utilizes the processing power of the destination system (often a data lake or cloud-based data warehouse) to perform transformations.
  - **When to Use**: Suitable for big data environments where the destination system can handle large-scale transformations. Beneficial when leveraging cloud-native services like AWS Redshift or Google BigQuery.
  - **Interview-Critical Distinction**: ELT is often more efficient for large datasets due to parallel processing capabilities of modern data warehouses.

## Batch vs Micro-Batch vs Streaming Trade-offs

- **Batch Processing**:
  - **Definition**: Data is collected over a period and processed in bulk.
  - **Underlying Mechanism**: Suitable for non-time-sensitive data processing tasks. Often involves scheduled jobs.
  - **When to Use**: Ideal for scenarios where data latency is not a concern, such as end-of-day reporting.
  - **Red Flag**: Confusing batch with real-time processing indicates a lack of understanding of latency requirements.

- **Micro-Batch Processing**:
  - **Definition**: A hybrid approach where data is processed in small, frequent batches.
  - **Underlying Mechanism**: Offers a balance between batch and streaming, reducing latency while maintaining efficiency.
  - **When to Use**: Suitable for scenarios where near-real-time processing is required but true streaming is not feasible.
  - **Complexity Bound**: Often involves trade-offs in terms of processing overhead vs. latency.

- **Streaming Processing**:
  - **Definition**: Continuous processing of data as it arrives.
  - **Underlying Mechanism**: Utilizes frameworks like Apache Kafka or Apache Flink to process data in real-time.
  - **When to Use**: Essential for applications requiring immediate insights, such as fraud detection or live analytics.
  - **Interview-Critical Distinction**: Candidates should understand the trade-offs in terms of system complexity and resource usage.

## Apache Airflow

- **DAG Definition**:
  - **Definition**: Directed Acyclic Graph representing a workflow of tasks.
  - **Underlying Mechanism**: Ensures tasks are executed in a specific order without cycles.
  - **When to Use**: Essential for defining complex workflows with dependencies.

- **Operators**:
  - **PythonOperator**: Executes Python functions.
  - **BashOperator**: Executes bash commands.
  - **Sensors**: Waits for a certain condition to be met before proceeding.
  - **Red Flag**: Misunderstanding the role of sensors can lead to inefficient task scheduling.

- **Scheduling**:
  - **Definition**: Defines when and how often a DAG should run.
  - **Underlying Mechanism**: Uses cron expressions or preset intervals.
  - **When to Use**: Critical for time-sensitive workflows.

- **XCom**:
  - **Definition**: Cross-communication mechanism for sharing data between tasks.
  - **Underlying Mechanism**: Stores small amounts of data in the Airflow metadata database.
  - **When to Use**: Useful for passing data between tasks without external storage.

- **Task Dependencies**:
  - **Definition**: Defines the order of task execution.
  - **Underlying Mechanism**: Uses upstream and downstream relationships.
  - **When to Use**: Ensures tasks are executed in the correct sequence.

- **Backfill**:
  - **Definition**: Running a DAG for past dates to fill in missing data.
  - **Underlying Mechanism**: Re-executes tasks for specified historical dates.
  - **When to Use**: Essential for data consistency and completeness.

## CDC (Change Data Capture)

- **Log-Based CDC**:
  - **Definition**: Captures changes by reading database logs.
  - **Underlying Mechanism**: Utilizes database transaction logs to detect changes.
  - **When to Use**: Suitable for high-volume databases where minimal impact on performance is required.

- **Trigger-Based CDC**:
  - **Definition**: Uses database triggers to capture changes.
  - **Underlying Mechanism**: Triggers execute custom logic on data changes.
  - **When to Use**: Useful for databases without native log-based CDC support.
  - **Red Flag**: Overuse of triggers can lead to performance degradation.

- **Debezium**:
  - **Definition**: An open-source platform for CDC.
  - **Underlying Mechanism**: Integrates with Kafka to stream database changes.
  - **When to Use**: Ideal for building real-time data pipelines with minimal latency.

## Data Quality Checks in Pipelines

- **Great Expectations**:
  - **Definition**: A framework for defining and executing data quality tests.
  - **Underlying Mechanism**: Uses expectations to validate data against predefined rules.
  - **When to Use**: Essential for ensuring data integrity and reliability.

- **dbt Tests**:
  - **Definition**: Data build tool tests for validating data transformations.
  - **Underlying Mechanism**: Executes SQL-based tests on transformed data.
  - **When to Use**: Suitable for validating data models and transformations in ELT pipelines.

## Error Handling and Retry Strategies

- **Error Handling**:
  - **Definition**: Mechanisms to manage and recover from errors in data pipelines.
  - **Underlying Mechanism**: Includes try-catch blocks, error notifications, and fallback procedures.
  - **When to Use**: Critical for maintaining pipeline robustness and reliability.

- **Retry Strategies**:
  - **Definition**: Techniques to reattempt failed tasks.
  - **Underlying Mechanism**: Configurable retry intervals and limits.
  - **When to Use**: Essential for transient errors and network issues.

## Idempotency Design

- **Definition**: Ensures repeated execution of a task yields the same result.
- **Underlying Mechanism**: Utilizes unique identifiers and state checks to prevent duplicate processing.
- **When to Use**: Critical for ensuring data consistency in distributed systems.
- **Interview-Critical Distinction**: Candidates should understand how idempotency prevents data corruption in retries.

## Pipeline Monitoring and Alerting

- **Monitoring**:
  - **Definition**: Continuous observation of pipeline performance and health.
  - **Underlying Mechanism**: Uses metrics, logs, and dashboards to track pipeline status.
  - **When to Use**: Essential for proactive issue detection and resolution.

- **Alerting**:
  - **Definition**: Notification mechanisms for pipeline failures or anomalies.
  - **Underlying Mechanism**: Configurable alerts based on thresholds and conditions.
  - **When to Use**: Critical for timely response to pipeline issues.
