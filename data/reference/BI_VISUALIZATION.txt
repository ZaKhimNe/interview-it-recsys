## Power BI

- **DAX Measures vs Calculated Columns**
  - **DAX Measures**: Calculations performed on data at the time of query execution. They are dynamic and context-sensitive, meaning their results can change based on the filters applied in the report.
    - **Underlying Mechanism**: Measures are computed using DAX (Data Analysis Expressions) and are evaluated in the context of the report's filters and slicers.
    - **When/Why It Matters**: Measures are optimal for aggregations and calculations that need to be dynamic and responsive to user interactions. They are stored in the model metadata and do not increase the data model size.
    - **Common Confusion**: Measures are often confused with calculated columns. The key difference is that measures are evaluated at query time, while calculated columns are evaluated at data load time.
  - **Calculated Columns**: Computed during data load and stored in the data model. They are static and do not change with report filters.
    - **Underlying Mechanism**: Calculated columns are added to the data model and stored in memory, increasing the model size.
    - **When/Why It Matters**: Useful for calculations that need to be consistent across all reports and do not depend on user interactions.
    - **Interviewer Red Flag**: Confusing measures with calculated columns, especially in scenarios requiring dynamic calculations.

- **Data Model and Relationships**
  - **Data Model**: The structured representation of data tables and their relationships within Power BI.
    - **Underlying Mechanism**: Power BI uses a columnar database engine (VertiPaq) to store and compress data efficiently.
    - **When/Why It Matters**: A well-designed data model ensures efficient queries and accurate results. It is crucial for performance optimization.
  - **Relationships**: Define how tables are connected in the data model, typically through primary and foreign keys.
    - **Types**: One-to-many, many-to-one, and many-to-many.
    - **When/Why It Matters**: Properly defined relationships ensure data integrity and accurate aggregations.
    - **Interviewer Red Flag**: Misunderstanding the impact of relationship cardinality on report performance and accuracy.

- **Star Schema in Power BI**
  - **Definition**: A data modeling design that organizes data into fact and dimension tables.
    - **Fact Tables**: Contain quantitative data for analysis (e.g., sales, revenue).
    - **Dimension Tables**: Contain descriptive attributes related to fact data (e.g., date, product).
  - **Underlying Mechanism**: Star schema simplifies queries and improves performance by reducing the number of joins.
  - **When/Why It Matters**: Essential for efficient data retrieval and analysis, especially in large datasets.
  - **Common Confusion**: Confusing star schema with snowflake schema. The key difference is that star schema has denormalized dimension tables, while snowflake schema has normalized dimensions.

## Tableau

- **LOD Expressions**
  - **Definition**: Level of Detail (LOD) expressions allow for calculations at different levels of granularity than the view.
    - **Types**: INCLUDE, EXCLUDE, and FIXED.
    - **Underlying Mechanism**: LOD expressions override the default query context to compute values at specified levels.
    - **When/Why It Matters**: Useful for complex aggregations and comparisons that require different levels of detail.
    - **Common Confusion**: Misunderstanding the difference between FIXED and INCLUDE/EXCLUDE. FIXED ignores the view context, while INCLUDE/EXCLUDE adjust it.

- **Calculated Fields**
  - **Definition**: Custom fields created using formulas to perform calculations on data.
    - **Underlying Mechanism**: Calculated fields are computed at the data source or in-memory, depending on the context.
    - **When/Why It Matters**: Essential for creating custom metrics and KPIs that are not directly available in the data source.
    - **Interviewer Red Flag**: Inability to distinguish between row-level and aggregate calculations in calculated fields.

- **Blending**
  - **Definition**: Combining data from multiple sources within Tableau.
    - **Underlying Mechanism**: Data blending uses a primary and secondary data source, linking them on common dimensions.
    - **When/Why It Matters**: Useful when data cannot be joined at the source level or when dealing with disparate data sources.
    - **Common Confusion**: Confusing blending with joining. Blending is performed at the visualization level, while joining is done at the data source level.

## Data Visualization Best Practices

- **Chart Selection**
  - **Definition**: Choosing the appropriate chart type based on the data and the message to be conveyed.
  - **When/Why It Matters**: Correct chart selection enhances clarity and comprehension. For example, use bar charts for categorical comparisons and line charts for trends over time.
  - **Interviewer Red Flag**: Selecting a chart type that misrepresents data or obscures insights.

- **Color**
  - **Definition**: Use of color to enhance or highlight information in visualizations.
  - **When/Why It Matters**: Effective use of color can draw attention to key insights and improve accessibility. Avoid using too many colors, which can confuse viewers.
  - **Common Confusion**: Overuse of color gradients, which can mislead interpretation of data magnitude.

- **Clarity**
  - **Definition**: Ensuring that visualizations are easy to understand and interpret.
  - **When/Why It Matters**: Clarity is crucial for effective communication of insights. Use clear labels, legends, and avoid clutter.
  - **Interviewer Red Flag**: Creating visualizations that are overly complex or lack necessary context.

## OLAP Concepts

- **Definition**: Online Analytical Processing (OLAP) is a category of software technology that enables analysts to extract and view business data from multiple perspectives.
  - **Underlying Mechanism**: OLAP systems use multidimensional data models, allowing for complex analytical and ad-hoc queries with rapid execution time.
  - **When/Why It Matters**: OLAP is essential for business intelligence applications, enabling users to perform multidimensional analysis.
  - **Common Confusion**: Confusing OLAP with OLTP (Online Transaction Processing). OLAP is for analysis, while OLTP is for transaction processing.

## Slowly Changing Dimensions (SCD)

- **Definition**: A technique used in data warehousing to manage and track changes in dimension data over time.
  - **Types**: Type 1 (overwrite), Type 2 (add new row), Type 3 (add new column).
  - **Underlying Mechanism**: SCDs allow historical data to be preserved and analyzed, providing insights into how data changes over time.
  - **When/Why It Matters**: Critical for maintaining accurate historical records and supporting time-based analysis.
  - **Interviewer Red Flag**: Misunderstanding the implementation and use cases of different SCD types, especially in scenarios requiring historical accuracy.
